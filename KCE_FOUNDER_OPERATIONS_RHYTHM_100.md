# KCE Founder Operations Rhythm (100)

## Objetivo
Hacer más visible cómo debe leer el founder la operación diaria sin depender solo de dashboards o memoria.

## Idea central
KCE ya no debe operar como web + paneles sueltos. Debe leerse como una cadena:
- ventas
- continuidad
- soporte
- booking
- higiene operativa

## Ritmo recomendado
### Apertura
- mirar founder lanes
- decidir cierres y rescates del día
- dejar claro qué casos merecen presión directa

### Bloque medio
- proteger continuidad entre chat, contacto, deals, tickets y booking
- evitar reiniciar discovery si el contexto ya existe

### Cierre
- dejar owner, task y siguiente paso escritos
- no dejar casos valiosos “huérfanos”

## Lectura por tipo de caso
- catálogo / tours → mismo día
- plan personalizado → ≤12h
- booking / soporte sensible → ≤2h
- higiene / operador → operador
