# P14 — Deal Playback + Auto Winner Selection (A/B)

## Objetivo
1) **Auditoría operable por deal**: reproducir la historia del deal (timeline) para detectar exactamente dónde se rompe el funnel.
2) **Optimización automática**: cuando una plantilla tiene variantes (A/B), el sistema **prefiere la variante ganadora** usando datos reales (paid-rate), con guardas de muestras y caché para no saturar Supabase.

---

## Novedades

### 1) Timeline Playback (Admin → Deals → Deal)
En `/admin/deals/[id]`:
- Botón **Playback/Pausar**
- Step controls (◀/▶)
- Selector de velocidad (0.5x–4x)
- Resumen del paso actual (timestamp, tipo, título, detalle)
- Export JSON (`deal-<id>-timeline.json`) para auditoría y soporte

### 2) Auto Winner Selection (Templates)
En `src/lib/templates.server.ts`:
- Se calcula performance por variante leyendo `crm_outbound_messages`:
  - `sent`, `replied`, `paid`
- Se elige winner por **paid-rate** con:
  - ventana default: **30 días**
  - mínimo de muestras default: **30 envíos**
- Todo está cacheado (TTL ~10 min) para evitar consultas repetidas.

El render endpoint `/api/admin/templates/render` ahora siempre incluye:
- `seed` (si llega)
- `preferWinner=true` por defecto

### 3) Visibilidad de winners (Admin → Templates)
En `/admin/templates`:
- Columna **Winner (30d)** que muestra:
  - variante ganadora y paid%
- Fuente: `/api/admin/templates/perf-summary?days=30`

---

## Operación recomendada
- Revisa `/admin/revenue` para identificar etapas “stale”
- Abre deals clave y usa **Playback** para entender el bloqueo
- Ajusta copy en `/admin/templates`
- Deja que el sistema use winner selection y monitorea métricas
